# 项目整体优化路线图

## 背景

本项目处于 ERP/管理系统框架搭建阶段，后端功能已有雏形，前端仍在搭建核心交互与通用页面能力。当前阶段不兼容旧逻辑，优先让架构边界、接口契约和错误暴露机制稳定下来。

## 总体原则

- 先统一契约，再扩展功能。
- 前后端目录和概念保持领域、应用、单据一致。
- 后端失败语义必须显式，禁止用 `null` 掩盖问题。
- 前端以双 tabs 和组件注册表为核心，菜单只提供元数据，真实组件由前端白名单决定。
- 通用能力先抽象最小闭环，等真实页面落地后再继续提炼。

## 第一阶段：契约校准

- 修正文档与代码不一致的响应码、端口、校验命令。
- 后端统一接口语义：`listPage`、`detail`、`createNewData`、`save`、`submit`、`delete`。
- 后端统一错误策略：资源不存在、状态非法、无权限、参数非法必须抛异常。
- 基础设施禁止静默吞错，例如 Long 反序列化失败必须暴露为参数异常。
- 前端拆分校验命令：`lint` 只检查，`lint:fix` 才自动修复，增加 `format:check`。

验收标准：

- `AGENTS.md` 与实际代码约定一致。
- 前端可执行 `pnpm lint`、`pnpm format:check`、`pnpm build`。
- 后端可执行 `mvn compile`。

## 第二阶段：前端工作台骨架

- 建立 `src/domain/{领域}/{应用}/{单据}` 业务目录。
- 建立 `src/domain/common/page`，沉淀 `PageType`、`OperationType`、`BillStatus` 等基础类型。
- 建立 `src/domain/common/registry/componentRegistry.tsx`，作为菜单组件白名单，并通过 `pnpm gen:registry` 生成注册表。
- 工作台内部业务 tabs 保持 DOM 不卸载，只控制显隐。
- 菜单点击通过 `component` 业务键查注册表，不能任意动态加载后端字符串。

验收标准：

- 至少一个菜单可通过注册表打开真实页面或明确的占位组件。
- 列表页单实例、编辑页按单据 id 多实例、新增页 uuid 临时实例规则有统一实现位置。

## 第三阶段：单据生命周期

- 建立业务单据状态：`A=SAVED`、`B=SUBMITTED`、`C=AUDITED`、`D=CLOSED`。
- `createNewData` 不返回 id。
- 新增保存成功后，前端调用 `detail(id)` 回显，并把 uuid 临时 tab key 替换为真实 id tab key。
- `save` 负责新增和暂存修改，内部明确区分 `addNew` 与 `update`。
- `submit` 独立负责提交并推进状态。
- 建立 `BillStatusUtil` 作为保存/提交状态校验入口；是否补注解式 AOP 等真实单据落地后再判断。

验收标准：

- 新增、保存、提交、查看状态流转清晰。
- 后端状态校验始终读取数据库当前状态，不信任前端传入状态。

## 第四阶段：通用页面框架

- 抽取 `ListPage`：查询区、表格、分页、工具栏、第一列点击打开单据。
- 抽取 `EditPage`：标题区、按钮区、表单区、`OperationType` 能力控制。
- 抽取 `CustomPage`：配置页、报表、监控、SQL 控制台等非标准页面兜底。
- 字段少的单据支持居中 Modal 编辑形态。

验收标准：

- 新增一个标准单据页面时，只需要配置列、表单、接口和注册表。
- 需要具体 UI 设计时，先由用户打开参考网站，再按参考设计生成。

## 第五阶段：数据库基线

- 使用项目根目录 `db/migration` 作为 Flyway 脚本唯一源码位置，统一管理表结构、备注和初始化数据；Maven 构建时将脚本复制到后端类路径。
- 创建表必须包含表备注和字段备注。
- 初始化数据至少覆盖超级管理员、基础权限、应用、菜单。
- 数据库结构和必要初始化数据以 Flyway 迁移脚本为唯一权威来源，结构变更必须新增版本脚本。

验收标准：

- 新环境可通过脚本恢复基础框架数据。
- 表结构备注与项目字段规范一致。

## 当前执行顺序

1. 已补齐文档、校验脚本、组件注册表生成脚本。
2. 已建立前端 `domain/common` 最小架构骨架和工作台 tab 打开规则。
3. 已收敛后端管理接口静默失败、Long 反序列化、单据状态基础能力。
4. 下一步选择一个低风险页面作为通用页面框架试点。
