# Smart Manage 企业级架构优化计划

## 1. 文档目标

本文档用于指导 Smart Manage 在当前架构基础上持续演进为可实际使用、可长期维护、符合主流社区习惯的企业级项目。

当前阶段以架构内核和工程质量为重点，不以快速扩展业务模块、制作开源展示材料或完成示范单据为目标。所有改造均应保持边界清晰、分批实施、每批可编译验证，避免一次性大规模重构造成架构反复。

## 2. 已确认的架构决策

| 决策项 | 结论 | 当前状态 |
|---|---|---|
| 后端总体架构 | 保留按领域、应用、单据组织的模块化单体架构 | 已确认 |

## 3. 目标架构基线

### 3.1 后端请求链路

```text
Filter / Interceptor
        ↓
Controller
        ↓
公开 Service（查询、业务命令入口、BizLog）
        ↓
包内 TxService（事务、状态校验、聚合写入）
        ↓
Mapper
        ↓
PostgreSQL
```

各层职责固定如下：

- Controller：路由、权限注解、参数校验、调用 Service、返回 `Result<T>`。
- Service：单据唯一公开入口、查询、VO 转换、业务命令定义、`@BizLog`、高风险身份校验、委托 TxService。
- TxService：读取数据库当前状态、唯一性校验、乐观锁校验、聚合内写入、主明细保存和显式删除。
- Mapper：只承担数据访问，不承载业务状态推进和权限判断。
- GlobalExceptionHandler：统一转换系统异常、参数异常和业务异常，不在 Controller 内重复捕获。

### 3.2 前端状态边界

| 状态类型 | 负责技术 | 典型内容 |
|---|---|---|
| 服务端状态 | TanStack Query | 列表、详情、引用选择、菜单、权限、保存和删除命令 |
| 跨页面客户端状态 | Zustand | 应用页签、单据页签、当前应用、侧边栏、登录用户快照 |
| 页面临时状态 | Ant Design Form / React State | 表单编辑值、选中行、弹窗开关、未保存数据 |

禁止在 Zustand 中复制和长期维护 Query 返回的服务端数据，也禁止把正在编辑的 Form 数据写入 Query Cache。

## 4. 实施原则

1. 每一阶段只处理一个明确的架构主题，不夹带无关重构。
2. 项目仍处于架构阶段，不保留旧字段、旧接口或旧 JSON 字段的兼容代码。
3. 数据库、后端和前端涉及同一契约时必须在同一批次完成修改。
4. 每次后端修改至少执行 `mvn compile`。
5. 每次前端修改至少执行 `pnpm lint`、`pnpm format:check`、`pnpm build`。
6. 当前阶段不以测试覆盖率作为验收条件，但不得降低已有编译和静态检查标准。
7. 关键设计应及时同步到 `AGENTS.md`，详细设计原因保留在 `docs/architecture`，避免 `AGENTS.md` 无限膨胀。

## 5. 分阶段优化计划

### 当前实施基线（2026-07-21）

- Flyway 已成为数据库结构和必要初始化数据的唯一权威来源，脚本位于项目根目录 `db/migration`。
- 乐观锁已统一为 `version`，启用状态已统一为 `enabled`。
- 用户、角色、权限、云、应用、菜单等核心资料已按公开 Service 与包级 TxService 分离事务。
- 启停、角色分配、权限分配已从普通保存中拆为独立命令。
- 前端已建立 ListPage、EditPage、AssignmentPage 和 CustomPage 页面边界。
- 本轮继续完善授权变更后的会话失效、禁用资源导航过滤、统一命令反馈和页面错误边界。
- 自动化测试体系仍按既定决策暂缓，不属于本轮范围。

### 阶段一：建立 Flyway 数据库迁移基线

#### 目标

将数据库结构从人工维护的快照方式迁移为可排序、可追踪、不可随意修改历史的版本化迁移体系，为后续 `mutex → version` 和其他结构调整提供可靠基础。

#### 工作内容

- 在后端引入 Flyway PostgreSQL 依赖。
- 在 `application.yml` 中配置所有环境共用的 Flyway 规则。
- 数据源地址、账号、密码等环境差异继续由环境配置文件覆盖。
- 以当前实际数据库结构为依据，审计并生成第一版基线迁移。
- 将必要系统数据与普通演示、运行数据分离。
- Flyway 生效后，禁止修改已经执行过的版本脚本；任何结构调整必须新增迁移版本。
- 不自动启用危险的 `baseline-on-migrate` 兜底。现有开发数据库应在人工确认后执行一次基线或重建。

#### 建议目录

```text
db/migration/
├── V1__baseline_schema.sql
├── V2__baseline_system_data.sql
└── V3__rename_mutex_to_version.sql
```

项目根目录仅保留 `db/migration` 这一套 Flyway 源脚本，不再保留 `db/postgresql` 结构快照；Maven 构建时将迁移脚本复制到后端类路径，这些复制文件仅属于构建产物。

#### 验收标准

- 空数据库能够按顺序执行全部迁移。
- 迁移后数据库结构与当前 Entity、Mapper 一致。
- 表和字段均保留中文注释。
- 后端能够正常编译和启动。
- 新增结构变更只能通过新的 Flyway 版本完成。

### 阶段二：统一乐观锁字段为 `version`

#### 目标

将语义容易混淆的 `mutex` 全链路改为社区通用的 `version`，并补齐真正的端到端并发控制。

#### 工作内容

- 通过 Flyway 将相关表的 `mutex` 字段重命名为 `version`。
- 字段类型继续使用 `integer NOT NULL DEFAULT 0`。
- 字段注释统一为“乐观锁版本号”。
- 所有可编辑聚合根 Entity 使用：

```java
@Version
private Integer version;
```

- Form、VO、前端类型和保存参数统一改为 `version`。
- `detail` 必须返回 `version`。
- 修改保存必须携带从详情获得的 `version`。
- 新增不要求前端提供 `version`，由数据库默认值和实体规则初始化。
- TxService 必须检查 `updateById` 返回值；返回 0 时抛出 `DATA_CONFLICT`。
- Entry、关系表、日志和执行记录不独立增加 `version`，由聚合根统一保护。

#### 涉及范围

当前至少涉及：

- App
- BasicData
- Cloud
- SysParam
- Permission
- Role
- User

目前部分聚合只有 Entity 存在乐观锁字段，但 Form、VO 和前端未形成闭环，本阶段需要一次完成，不能只做字段改名。

#### 验收标准

- 项目内不再出现业务字段名 `mutex`。
- 所有可编辑聚合根详情和保存契约包含 `version`。
- 所有修改写入检查影响行数。
- 并发冲突统一返回 `DATA_CONFLICT`，不静默覆盖数据。
- 数据库、后端、前端命名完全一致。

### 阶段三：完善统一异常和错误码体系

#### 目标

确保调用方能够根据错误类型采取明确行为，避免所有业务失败都被归类为服务器异常。

#### 工作内容

- 逐步禁止 `new BizException("...")` 这种无法表达错误类别的构造方式。
- 业务异常必须显式指定 `ResultEnum`。
- 按实际需要完善以下错误类别：
  - 参数错误
  - 未登录
  - 权限不足
  - 资源不存在
  - 数据并发或唯一性冲突
  - 单据状态错误
  - 系统配置错误
  - 外部服务调用失败
  - SQL 或持久化异常
- GlobalExceptionHandler 补齐常见框架异常映射：
  - JSON 格式错误
  - 参数绑定错误
  - Bean Validation 错误
  - 数据唯一约束冲突
  - 外键约束冲突
  - 乐观锁冲突
  - 文件上传超限
- 对外响应不返回堆栈、SQL 或敏感配置，服务端日志保留完整异常并关联 `traceId`。
- 删除或收紧暂未实现却对外公开的方法，禁止运行时抛出 `UnsupportedOperationException` 作为正常业务能力占位。

#### 验收标准

- 公开 Service 不再使用无类型的业务异常表达可预期失败。
- 相同错误场景返回相同错误码。
- 唯一约束、外键约束和乐观锁冲突具有可区分提示。
- 前端可以根据错误码处理并发冲突、登录失效等特殊场景。

### 阶段四：统一权限与高风险能力边界

#### 目标

让每个接口的访问级别清晰可审计，避免依靠“是否漏写权限注解”表达访问策略。

#### 工作内容

- 将接口明确分为：
  - `PUBLIC`：验证码、登录等匿名能力。
  - `AUTHENTICATED`：所有已登录用户可用的公共能力。
  - `PERMISSION`：必须拥有具体业务权限码。
- 审计所有 Controller 方法并明确归类。
- 审计免登录路径，重点检查上传、Druid、API 文档和监控端点。
- SQL 控制台、脚本控制台、Arthas 继续保留完整能力，但必须在公开 Service 入口校验当前账号确实为 `administrator`。
- 高风险操作必须记录 `@BizLog`，不得记录密码、令牌、私钥和完整敏感配置。
- 权限码命名统一遵循 `{domain}:{app}:{bill}:{action}`，不在同一模块混用不同层级。

#### 验收标准

- 每个 Controller 接口都可以明确归入一种访问类型。
- 普通用户即使被误配高风险业务权限码，也无法调用管理员专属能力。
- 免登录列表中不存在未经过明确确认的写接口和管理接口。
- 高风险操作日志可追踪且不泄露敏感信息。

### 阶段五：完善事务、缓存和外部系统一致性

#### 目标

明确数据库事务不能覆盖 Quartz、Redis、FTP 和本地文件系统，避免形成虚假的原子性保证。

#### 工作内容

- 审计所有 Service，写操作必须委托包内 TxService。
- 数据库写入必须检查 `insert`、`updateById`、`deleteById` 的结果。
- 建立缓存失效矩阵，明确每个写操作需要失效的缓存。
- 缓存失效在数据库事务成功提交后执行。
- 定时任务以数据库状态为业务依据，明确 Quartz 同步失败后的重试或重新同步入口。
- 文件和 FTP 操作明确临时文件、正式文件、数据库记录三者的失败处理顺序。
- 异步日志写入失败不得影响主业务，但必须留下可观测的告警。
- 当前阶段不引入消息队列、分布式事务或 Outbox；只有实际业务出现可靠异步需求时再评估。

#### 验收标准

- 不存在公开 Service 直接执行普通业务 Mapper 写入。
- 事务成功后缓存能够正确失效。
- Quartz、文件、FTP 操作失败时不会留下无法识别的业务状态。
- 多次重复执行可恢复操作不会产生重复主数据。

### 阶段六：重构前端 TanStack Query 使用方式

#### 目标

让 TanStack Query 真正负责服务端状态和缓存同步，而不只是替代按钮 loading。

#### 工作内容

- 每个单据建立 Query Key Factory，禁止继续散落手写字符串 key。
- Query Key 至少区分：
  - all
  - lists
  - list(params)
  - details
  - detail(id)
- 保存、删除、提交等 Mutation 由具体领域页面或领域 Hook 持有。
- 通用 `EditPage` 和 `ModalEditPage` 保持 UI 职责，只接收 `saving`、`onSave` 等属性。
- 保存成功后按业务需要：
  - 失效列表缓存。
  - 更新或重新获取详情。
  - 新增页替换临时 tab key。
  - 使用真实 ID 重新调用 `detail`。
- 删除成功后清理对应详情缓存并失效列表。
- 角色、用户、菜单等关联变更需要同时失效权限和菜单缓存。
- 高风险的一次性执行结果不进入长期 Query Cache。

#### 验收标准

- 业务页面不再散落无规则的 Query Key 字符串。
- Mutation 成功后相关列表、详情和树不会展示旧数据。
- Query 数据不复制到 Zustand。
- 页面不存在为了请求数据而手工编写的 `useEffect + useState` 缓存逻辑。

### 阶段七：完善前端公共组件和类型体系

#### 目标

降低公共组件对类型断言和隐式约定的依赖，使新增业务页面能够获得可靠的 TypeScript 约束。

#### 工作内容

- 重构 RefSelector 泛型，使 fetch、columns、displayRender、value 和 onChange 使用同一记录类型。
- 清理业务页面中的 `as unknown as` 双重断言。
- 统一通过 Ant Design `App.useApp()` 获取 message、modal 等上下文能力。
- 统一 EditPage 与 ModalEditPage 的字段渲染逻辑，但保留两种不同页面布局。
- 保持 Ant Design Form 作为表单校验唯一来源，不重新引入 Zod。
- 继续执行 TSX 禁止内联 CSS、自定义类名 `sm-` 前缀等现有规范。
- 页签关闭、退出登录和应用关闭统一经过脏数据检查。

#### 验收标准

- 普通业务页面不需要通过双重断言接入 RefSelector。
- 保存、删除等操作统一展示上下文消息和提交状态。
- 公共字段渲染逻辑不存在多份行为不一致的实现。
- `pnpm lint`、`pnpm format:check`、`pnpm build` 全部通过。

### 阶段八：整理配置边界与可观测性

#### 目标

形成清晰的公共配置、环境差异配置和敏感配置边界，同时提升生产问题定位能力。

#### 配置约定

- `application.yml`：所有环境共用的应用名称、序列化、MyBatis-Plus、日志格式、响应压缩、公共安全规则和通用业务默认值。
- `application-dev.yml`：本地开发端口、开发数据源、开发 Redis、开发 CORS、开发调试能力。
- `application-prod.yml`：生产数据源连接方式、生产 Redis、日志级别、监控暴露策略和生产安全差异。
- 公共配置使用 `spring.profiles.active=${SPRING_PROFILES_ACTIVE:dev}`，开源项目默认进入开发环境；部署时必须通过外部环境变量显式选择 `prod`，切换环境不修改仓库配置。
- `application-dev.yml` 可提供明确标注、不可用于生产的公开开发凭据和开发密钥，使开源项目可直接启动；`application-prod.yml` 中的密码、私钥、令牌仍必须通过环境变量或外部配置注入。

#### 可观测性工作

- 统一日志级别和日志结构。
- 所有异常响应带 `traceId`。
- 操作日志、登录日志和系统日志职责分离。
- Actuator 仅暴露必要端点，生产环境不公开详细组件和内部配置。
- Druid、SQL、脚本和 Arthas 等管理能力采用独立安全边界。

#### 验收标准

- 公共配置和环境差异不存在重复、冲突定义。
- 切换 profile 不需要修改公共配置文件。
- 仓库内不存在可直接用于正式环境的私钥和密码。
- 生产监控端点不会匿名暴露敏感运行信息。

### 阶段九：持续代码规范治理

#### 目标

将现有约定变成可持续执行的代码基线，而不是依靠开发者记忆。

#### 当前保持不变的命名

本轮只实施 `mutex → version`。以下名称继续保留，未经新的架构决策不得顺带修改：

- `entrys`
- `ADDNEW`
- `createNewData`
- Service 的 `deleteById`
- 业务编码字段 `number`

#### 持续检查项

- TxService 必须包级可见。
- Controller 和其他单据不得导入 TxService。
- `@BizLog` 只能出现在公开 Service 命令方法。
- 公开 Service 不使用 `return null` 表达业务失败。
- Java 和 TypeScript 禁止无意义的单字母变量名。
- XML Mapper 主表别名使用 `a`，后续 JOIN 使用 `b`、`c`、`d`。
- 后端 Long ID 在前端始终使用 string。
- 删除无调用方的未实现公开方法，不保留兼容占位。

## 6. 暂缓事项及启动条件

### 6.1 自动化测试体系

当前不修改“项目处于架构搭建阶段，不扩展测试”的约定。

满足以下条件后启动测试体系建设：

- Flyway 基线稳定。
- 异常和错误码体系稳定。
- `version` 乐观锁闭环完成。
- 权限接口分类完成。
- 至少一批核心 Service 和前端数据流不再频繁调整。

届时优先建设：

- 应用启动和配置加载验证。
- Flyway 空库迁移验证。
- 权限与超级管理员安全边界。
- 乐观锁并发冲突。
- 主明细聚合事务。
- 登录、验证码和加解密链路。

不设机械覆盖率目标，不为简单 Getter、Setter 和无业务分支的普通转换方法编写低价值测试。

## 7. 推荐执行顺序

| 顺序 | 阶段 | 前置依赖 |
|---|---|---|
| 1 | Flyway 数据库迁移基线 | 无 |
| 2 | `mutex → version` 与乐观锁闭环 | Flyway 基线 |
| 3 | 统一异常和错误码 | 可与阶段 2 分批并行，但不要混在同一次修改中 |
| 4 | 权限与高风险能力边界 | 异常体系基本稳定 |
| 5 | 事务、缓存和外部系统一致性 | 权限与异常规则明确 |
| 6 | TanStack Query 数据流 | 后端详情、保存和错误契约稳定 |
| 7 | 前端公共组件和类型体系 | Query 数据流基本稳定 |
| 8 | 配置边界与可观测性 | 可按模块穿插实施 |
| 9 | 自动化测试体系 | 核心基础契约稳定后启动 |
| 10 | 开源展示、示范单据和社区建设 | 架构趋于稳定后启动 |

## 8. 每阶段交付要求

每个阶段完成时必须提供：

1. 本阶段解决的问题和未解决的问题。
2. 实际修改范围。
3. 数据库、后端、前端契约变化说明。
4. 对 `AGENTS.md` 的同步修改。
5. 编译、Lint、格式和构建结果。
6. 下一阶段是否具备启动条件。

如果实施过程中发现某项修改会改变本文已确认的决策，必须暂停并重新确认，不得以“顺便优化”为由扩大改动范围。
