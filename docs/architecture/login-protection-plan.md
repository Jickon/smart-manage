# 登录保护演进方案

## 状态

本方案在采购申请参考模块完成打磨后实施。当前只记录设计边界，不提前加入无界面的动态参数和运行逻辑。

## 目标

登录保护同时限制账号维度和客户端 IP 维度的异常请求，降低密码猜测、验证码滥用和单账号攻击风险。所有计数使用 Redis 保存，登录日志继续承担审计职责。

## 建议策略

- 账号失败窗口：统计指定时间窗口内同一用户名的失败次数。
- IP 失败窗口：统计指定时间窗口内同一客户端 IP 的失败次数。
- 短时锁定：任一维度达到阈值后，在锁定时间内拒绝继续登录。
- 验证码频率：限制同一 IP 获取验证码的最小间隔和每分钟上限。
- 成功清理：登录成功后清除该账号的失败计数，IP 计数按窗口自然过期。
- 统一反馈：避免通过错误信息暴露用户名是否存在，详细原因只写入认证日志。

## 配置边界

YAML 保存不能被普通管理员削弱的安全底线：

- 登录保护总开关的生产默认值；
- 验证码校验要求；
- 失败日志要求；
- 各阈值允许调整的最小值和最大值；
- Redis 不可用时的失败策略。

系统参数保存允许运维动态调整的阈值：

- 账号和 IP 失败次数；
- 统计窗口；
- 锁定时长；
- 验证码请求间隔及每分钟上限。

动态参数读取失败时使用 YAML 中的安全默认值并记录告警，禁止静默关闭保护。

## 实施前置条件

- 系统参数管理前端完成；
- 登录保护参数具备明确的管理权限；
- 客户端 IP 获取规则完成反向代理可信边界配置；
- Redis key、过期时间及认证日志脱敏规则完成评审；
- 补充账号/IP 限制、自动解锁、成功清理和 Redis 故障策略测试。
