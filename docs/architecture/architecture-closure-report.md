# 架构收口报告

## 收口范围

本轮以现有架构契约为准，不新增分层和运行时插件机制，处理以下边界：

- 公开 Service 与包级 TxService 的事务分离
- 业务异常、持久化结果和乐观锁冲突
- Controller 权限与高风险管理员身份校验
- 事务提交后的缓存失效
- 前端 Query Key、命令反馈、单据页签与脏数据检查
- Flyway、后端编译和前端静态检查门禁
- 使用采购申请验证标准主从业务单据

## 审计结论

- 普通领域写操作已经集中在 `*TxService`；认证、缓存清理和异步日志属于明确的基础设施例外。
- `@Transactional` 均位于事务服务，没有发现公开业务 Service 直接声明事务。
- 业务异常均显式携带 `ResultEnum`，没有发现无类型 `BizException`。
- SQL 执行日志的数据库写入原先未检查影响行数，本轮已收紧。
- SQL、脚本和 Arthas 的公开入口保留 `administrator` 身份校验。
- 基础数据等缓存已经在数据库事务提交后失效。
- 前端标准编辑页原有关闭回调基础设施，但业务页面未注册；本轮由 `EditPage` 统一接入。
- 公共编辑页已支持同一 Form 内承载主从明细，不为采购申请建立专用页面框架。

## 采购申请验收范围

采购申请覆盖：`createNewData`、新增、暂存修改、提交、只读查看、删除、主从聚合、乐观锁、权限、操作日志、Query 缓存失效、临时页签替换和脏数据离开确认。

暂不实现审核、工作流、消息通知、采购订单转换、物料主数据和通用编码引擎。

## 进入业务扩展阶段的判定

只有以下检查全部通过后，才判定具备有序扩展条件：

1. `mvn compile`
2. `pnpm lint`
3. `pnpm format:check`
4. `pnpm build`
5. 空数据库顺序执行全部 Flyway 迁移
6. 采购申请保存、提交、并发冲突和页签生命周期的人工验收

上述检查及采购申请纵向人工验收均已完成。项目已具备“受控业务扩展”条件：后续可以一次扩展一个业务聚合，并复用采购申请所验证的架构路径；暂不进入多模块并行铺量阶段。
